Payza Style Guide

Style used at Payza

View the Project on GitHub

Overview

This document contains guidelines and coding standards for web applications built by the Payza UI Development Team.

Goal

  • Separate presentation, content, behavior
  • Semantic, well-formed, valid mark-up
  • Enhance user experience (using javascript)

CSS

General Practices

  • Use external CSS. It is forbidden to use CSS inline of HTML tags and CSS embedded in HTML.
  • Avoid shorthand if it's hard to read. Example: font, background, list-style, etc.
  • Always use semi-colon (;) at the end of the each property even if there's only one property declared.
  • Always use lowercase when creating selectors (HTML tags, CLASS, ID) or adding a value to a color property (hex, rgba)
  • Use class for styling; ids are reserved for javascript.
  • Sprite-based selectors must be grouped and organized alphabetically. Example: When using icon flags to represent a country or currency.
  • Readability should be taken priority over compression.
  • Write efficient CSS
  • Remember CSS Specificity when creating or trying to overwrite a property. Make sure that the cascading nature of CSS is understood clearly.
  • Selectors are case-sensitive.
  • Multiple classes can be assigned to a single element.

Syntax

Default format

The default format when declaring CSS rules is as follows: selector — space — opening curly brace { — CSS property — colon : — space — value — semi-colon ; — closing curly brace }.

Single Line Declaration

Single line declaration can only be used when there's only one property declared.

selector {property: value;}
Multi-line Declaration

Each property-value pair should be on a single line, similar for multiple selectors. Group selectors alphabetically, starting with HTML elements, .classes and lastly #ids. Only use the Tab key for indentation. A tab is equal to 2 spaces.

p, .class, #id {
  property: value; 
  property: value;}

Shorthand vs. Independent Declaration

Usage of shorthand properties is not permitted except:

  • When all 4 sides of a property is set to the same value 1
  • When setting the background image 2
ul {
  background: url(img/background.png) no-repeat top left; 2
  margin: 10px;} 1

The Order of Properties

CSS properties must be ordered alphabetically unless otherwise required for overwrite.

p {
  background-color: #ccc; 
  color: #000; 
  font-size: 1.3em;}

As much as possible, avoid shorthand. Only use shorthand if it's not hard to read/understand 1 or if the value of all 4 sides are the same 2. Do not use shorthand for font, list-style, outline properties.

p {
  background: #ccc; 
  border: 1px solid #000; 1
  color: #000; 
  font-size: 1.3em;
  margin: 10px;}2

If any of the individual side requires a different value, use independent declaration.

p {
  background: #ccc; 
  color: #000; 
  font-size: 1.3em;
  margin-top: 10px; margin-right: 20px;}

If a shorthand property is used, separate the independent properties set on a single line.

p {
  background: #ccc; 
  color: #000; 
  margin: 0;
  margin-top: 10px; margin-right: 20px;}

Handling Vendor-specific Properties using Browser Prefixes

The rule of thumb is that CSS properties be grouped accordingly and organized alphabetically. That being said, CSS2 properties must be separated from CSS3 properties for efficiency. CSS3 properties that require vendor-specific prefixes must be written starting with Mozilla Firefox (-moz-), Safari/Webkit (-webkit-), Opera (-o-), IE (-ms-) and the W3C Recommendation last.

p {
  color: #000; 
  background: #ccc; 
  font-size: 1.3em;
  /* CSS3 */
  -moz-border-radius: 3px; -webkit-border-radius: 3px; border-radius: 3px;
  -moz-box-shadow: 10px 10px 5px #888; -webkit-box-shadow: 10px 10px 5px #888; box-shadow: 10px 10px 5px #888;}

Naming Selectors

A key point to remember is to ask the question "What is the element for?". It is preferrable to choose names based on the functionality and not the appearance. Use names that are descriptive and intuitive. Avoid unnecessary long names and abbreviations. Always write in full English descriptions.

.container { ... } 
---
.note { ... } not .blue-text { ... }
---
.article-title { ... } not .news-article-main-title { ... }
---
.first-name { ... } not .f-name { ... }

A class .external-link is a more appropriate name than .blue-link for outbound links because when the visual style changes, it still makes sense.

.external-link {color: #c00; /* Red */}
--
.blue-link {color: #369; /* Blue */}

Distinction between CLASSes and IDs

Here are some key points to remember:

  • Avoid names that are very similar.
  • Only give an element ID if it is unique.
  • Must be lowercase
.notification { ... }
#notification_id { ... }                
--- not ---                
.notification { ... }
#notification { ... }

Colors

Color values are written in hexadecimal or rgba format in lowercase. The usage of color names (aqua, green, red, etc.) in plain text is not permitted, except for transparent. If an element doesn't call transparency use the hexadecimal format. Otherwise, use the rgba format.

p {color: #444;}
.note {border-color: rgba(0, 0, 0, 0.5);}

Commenting

Multi-line comments isn't necessary because CSS is straightforward to read and understand when organized properly. It's not necessary to use commenting more than described in this guideline. Use comments /* */ when:

  • Grouping blocks of selectors, on a single line 1
  • Grouping properties or separating CSS2 properties from CSS3 properties, on a single line 2
  • Commenting a trick, after the closing semi-colon before the closing curly braces 3
html {overflow-y: scroll; /* Vertical Scrollbar */3}
/** Layout **/ 1
header { ... }                
.container { ... }
footer { ... }                
a {
  background-color: #369;
  color: #fff;
  /* CSS3 */ 2
  -moz-border-radius: 6px; -webkit-border-radius: 6px; border-radius: 6px;}

URLs

Quotes (single / double) are not required when including a URL, for instance, background image.

.icon {background-image: url(img/sprite.png);}

Fonts

It is recommended to quote font family names that contain white space, digits, or punctuation characters other than hyphens and font family names that are included using the @font-face rule. Font family names may be case-sensitive on some operating systems.

@font-face {
  font-family: "OpenSansBold";
  font-style: normal;
  src: OpenSans-Bold-webfont.eot;
...
h1 {font-family: "OpenSansBold", "Trebuchet MS", Helvetica, sans-serif;}

Browser Hacks

When all else fails, writing hacks should be the very last resort. To identify what browser is being used, at least for IE browsers, conditional statements may be used. For other browsers, an option considered is using javascript to detect the browser. The idea is to insert a class into the html of the document. The class value will be the CLASS selectors representing the browser version and/or type.

<html class="ie8">

Other Things to Consider

  • Zero-in defaults using Reset or Normalize CSS
  • YUI Fonts CSS to normalize font size for all browsers
  • Clear floating elements using Clearfix
  • CSS rule grouping level (default, global, block, section, page-specific)
